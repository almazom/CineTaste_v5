# Золотой стандарт CineTaste

Версия: 1.1
Статус: активный R&D baseline
Область: внутренний архитектурный стандарт проекта CineTaste
Нормативный артефакт: этот Markdown-файл (`docs/golden-standard.md`)
Публикуемый артефакт: `docs/golden-standard.html` должен быть семантически эквивалентен
Последняя сверка с источниками: PROTOCOL v5.4.0 + FLOW v1.3.0 (2026-03-04)

---

## 1. Миссия

CineTaste — это внутренняя архитектурная лаборатория.

Главная цель — не масштабирование потребительского продукта.
Главная цель — проверка и укрепление золотого стандарта AI-assisted разработки, основанного на:

1. executable markdown-оркестрации,
2. изолированных CLI-обёртках,
3. строгих контрактах, протоколах и конвенциях,
4. детерминированной дисциплине планирования и верификации.

Пользовательская польза важна, но вторична по отношению к архитектурной чистоте и воспроизводимости.

## 2. Принципы золотого стандарта

### Принцип A: Executable Markdown — центр управления

`FLOW.md` — это не просто документация. Это исполняемый, версионируемый артефакт оркестрации.

Обязательные свойства:

- Каждая версия пайплайна явная (`flows/vX.Y/FLOW.md`).
- `flows/latest` указывает на активную версию.
- Runtime-исполнение следует flow-спецификации, а не неявным shell-договорённостям.
- Изменения вносятся через версионирование flow, а не «тихие» правки на месте.

Ожидаемый результат:

- Воспроизводимое поведение оркестрации.
- Простой rollback и сравнение версий flow.
- Единая операционная картина для агентов и людей.

### Принцип B: Всё, что можно изолировать, должно быть CLI

Каждое осмысленное действие должно быть выделено в отдельный CLI-инструмент в `tools/`.

Обязательные свойства CLI:

- Вызывается на уровне корня проекта (`ct-fetch`, `ct-schedule`, ...).
- Имеет явные аргументы и интерфейс `--help`.
- Имеет машиночитаемое описание API через `MANIFEST.json`.
- Проверяет заявленный входной контракт на entrypoint (если входной контракт объявлен).
- Использует детерминированную таксономию exit-кодов:
  - `0` — успех,
  - `2` — невалидные аргументы/путь,
  - `3` — сбой внешней зависимости,
  - `4` — ошибка парсинга/валидации контракта,
  - `1` — непредвиденная runtime-ошибка.
- Тестируется изолированно, без обязательного запуска всего пайплайна.

Ожидаемый результат:

- Компонуемые станции в цепочке пайплайна.
- Высокая наблюдаемость и низкая связность.
- Реальная заменяемость по контракту и интерфейсу.

### Принцип C: Contract-first и Protocol-first разработка

Подход implementation-first запрещён.

Обязательная последовательность:

1. конвенции и протокол,
2. схемы контрактов,
3. манифест API инструмента,
4. flow-спецификация,
5. имплементация,
6. верификация.

Это делает имплементацию заменяемой деталью за стабильными границами.

### Принцип D: Kanban JSON — planning SSOT

Перед имплементацией обязательно существует timestamped Kanban JSON.

Обязательные свойства:

- Путь: `.aura/kanban/KANBAN-YYYY-MM-DD-HHMM.json`.
- Symlink `latest` указывает на активный план.
- Статусы задач явные: `TODO`, `DOING`, `DONE`.
- Критерии приёмки объявлены до начала кодинга.

Ожидаемый результат:

- Детерминированные переходы состояний работы.
- Трассируемое намерение выполнения.
- Минимизация скрытых решений.

### Принцип E: Aura как runtime-дисциплина

`AURA.md` определяет, как работа исполняется в сессии.

Она управляет:

- порядком выполнения,
- ожиданиями по тестам и валидации,
- ограничениями поведения агентов,
- дисциплиной изменений.

Aura — это протокол исполнения, а не документ о продуктовой цели.

### Принцип F: Заменяемость через стабильный интерфейс

Если CLI сохраняет контракт и API, язык и внутренняя реализация могут меняться с минимальным влиянием на систему.

Это возможно потому, что связность опирается на:

- JSON-контракты,
- интерфейс уровня манифеста,
- вызов уровня flow,

а не на внутреннюю структуру кода.

### Принцип G: Паритет публикации и целостность доктрины

Markdown — нормативный источник доктрины. HTML — слой рендеринга.

Обязательные свойства:

- HTML не должен вводить нормативные правила, которых нет в Markdown.
- Семантика разделов Markdown и HTML должна совпадать.
- Любая правка доктрины сначала делается в Markdown, затем зеркалится в HTML.

Ожидаемый результат:

- Отсутствие доктринального дрейфа между исходником и публикацией.

## 3. Модель границ (Boundary Model): кто за что отвечает

### PROFILE.md (why)

Хранит intent проекта, ценности, ограничения и ожидания к review.
Определяет, что такое качество для проекта.

### AURA.md (how to work)

Хранит протокол выполнения и дисциплину работы в сессии.
Определяет, как люди и агенты должны выполнять задачи.

### PROTOCOL.json (what exists)

Хранит topology SSOT: контракты, инструменты, связи flow, версии.
Определяет статическую карту архитектуры.

### FLOW.md (what runs)

Хранит исполняемую последовательность оркестрации.
Определяет конкретный runtime-порядок шагов и цепочку команд.

### MANIFEST.json + contracts/*.schema.json (interfaces)

Хранят API-границы инструментов и данных.
Определяют безопасное взаимодействие частей системы.

Каноническое правило конфликтов:

- `PROFILE.md` приоритетен для intent и приоритетов,
- `AURA.md` приоритетен для дисциплины исполнения,
- `PROTOCOL.json` приоритетен для topology-фактов,
- `FLOW.md` приоритетен для runtime-порядка шагов и семантики команд.

## 4. Операционная модель изменений

Любое существенное изменение должно отвечать на вопросы:

1. Какой протокол/конвенция меняется?
2. Какая контрактная граница затронута?
3. Какой CLI-интерфейс затронут?
4. Какая версия flow затронута?
5. Как выполняется верификация?

Дополнительный обязательный gate:

- Проверка governance-consistency должна проходить до merge и до runtime-запуска.

Минимальный набор consistency-checks:

1. синхронизация порядка стадий (`PROTOCOL` vs `FLOW`),
2. синхронизация статусов active vs legacy инструментов,
3. синхронизация producer/consumer маппинга контрактов,
4. синхронизация version metadata.

Если эти проверки неявны, качество изменения ниже стандарта.

## 5. Ожидания к LLM в этом проекте

LLM должен оптимизировать сначала архитектурную целостность.

Порядок приоритетов:

1. сохранять целостность протоколов и контрактов,
2. сохранять CLI-изоляцию и компонуемость,
3. сохранять трассируемость версий flow,
4. и только потом оптимизировать удобство имплементации.

LLM не должен добавлять enterprise-сложность, которая не обслуживает внутреннюю R&D-цель.

Для review-loop использовать bounded sampling:

- 3–5 независимых проходов review для нестабильных задач,
- фокусные оптики в каждом проходе (security, contracts, reliability и т.д.),
- объединение находок через severity-triage,
- остановка по заданным quality-gates (без бесконечной полировки).

## 6. Критерии готовности (Definition of Done) для соответствия золотому стандарту

Изменение считается соответствующим, только если выполнено всё:

1. существует план в Kanban JSON,
2. затронутые границы явно зафиксированы,
3. contracts/manifests/flow обновлены до или вместе с кодом,
4. инструменты остаются независимо исполняемыми,
5. присутствуют верификационные артефакты,
6. проходит active-flow non-send E2E,
7. достигнут coverage threshold (`>= 80% total`),
8. приложены evidence-артефакты (минимум: сводка тестов, сводка покрытия, путь к run-артефактам).

## 7. Анти-паттерны

Следующие практики нарушают стандарт:

- implementation-first без обновления протоколов/контрактов,
- скрытое поведение вне flow и манифестов,
- shared mutable coupling между инструментами,
- недокументированное поведение CLI,
- неверсионируемые изменения оркестрации,
- устаревшие legacy-инструкции, противоречащие активной runtime-цепочке,
- противоречивые mode-semantics в управляющих документах.

---

## 8. Шкала критичности (Severity Taxonomy, P0-P4)

Используйте эту шкалу для review-находок и release-решений.

- `P0`: критический сбой корректности/безопасности; блокирует релиз.
- `P1`: высоко-значимый сбой в core-flow; исправляется до релиза.
- `P2`: существенный gap качества/надёжности; исправляется в текущем или следующем цикле.
- `P3`: умеренное улучшение; backlog при отсутствии немедленного риска.
- `P4`: опциональная полировка/стиль.

Базовое release-правило:

- релиз блокируется при наличии открытых `P0`/`P1`.

## 9. Итог

Золотой стандарт CineTaste — это markdown-центричная, contract-driven, CLI-компонуемая архитектурная дисциплина.

Система — это не только пайплайн рекомендаций, но и контролируемая среда для воспроизводимой AI-assisted инженерной практики.
