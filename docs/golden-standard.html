1. Миссия

CineTaste — внутренняя архитектурная лаборатория. Цель не в масштабировании потребительского продукта, а в валидации воспроизводимого золотого стандарта AI-assisted разработки.

  • Executable markdown-оркестрация
  • Изолированные CLI-обёртки
  • Строгие контракты/протоколы/конвенции
  • Детерминированная дисциплина планирования и верификации

Пользовательская польза важна, но вторична по отношению к архитектурной целостности и воспроизводимости.

2. Принципы золотого стандарта

Принцип A: Executable Markdown — центр управления

FLOW.md исполняемый и версионируемый; runtime следует flow-спецификации, а flows/latest выбирает активную версию.

Принцип B: Всё, что можно изолировать, должно быть CLI

Каждая отделяемая ответственность оформляется как отдельный CLI в tools/ с аргументами, --help, manifest API, проверкой входного контракта и детерминированной exit-таксономией (0/1/2/3/4).

Принцип C: Contract-first / Protocol-first

  1. Конвенции и протокол
  2. Схемы контрактов
  3. Манифесты API инструментов
  4. Flow-спецификация
  5. Имплементация
  6. Верификация

Принцип D: Kanban JSON — planning SSOT

Любая имплементация стартует с timestamped Kanban JSON и явных статусов TODO/DOING/DONE.

Принцип E: Aura как runtime-дисциплина

AURA.md управляет дисциплиной исполнения (порядок работ, требования к тестам, ограничения поведения агентов, дисциплина изменений).

Принцип F: Заменяемость через стабильный интерфейс

Внутренняя реализация (включая язык) может меняться, если контракт, CLI API и flow-вызов остаются стабильными.

Принцип G: Паритет публикации и целостность доктрины

Markdown — нормативный источник. HTML — рендеринг. HTML не должен вводить правила, которых нет в Markdown; правки делаются сначала в Markdown, затем зеркалятся в HTML.

3. Модель границ (Boundary Model)

  • PROFILE.md: intent проекта, ценности, ограничения.
  • AURA.md: протокол исполнения и дисциплина работы.
  • PROTOCOL.json: topology SSOT (контракты, инструменты, версии, flow map).
  • FLOW.md: исполняемая runtime-последовательность.
  • MANIFEST.json + schemas: стабильные интерфейсные границы инструментов и данных.
Каноническое правило конфликтов: PROFILE задаёт intent и приоритеты; AURA — дисциплину исполнения; PROTOCOL — topology-факты; FLOW — runtime-порядок и семантику команд.

4. Операционная модель изменений

Каждое изменение должно явно фиксировать протокол, контракт, CLI-интерфейс, версию flow и способ верификации.

Обязательный governance-gate: consistency-checks должны проходить до merge и до runtime-запуска.

  1. Синхронизация порядка стадий (PROTOCOL vs FLOW)
  2. Синхронизация active vs legacy статусов инструментов
  3. Синхронизация producer/consumer маппинга контрактов
  4. Синхронизация version metadata

5. Ожидания к LLM

  1. Сохранять целостность протоколов и контрактов
  2. Сохранять CLI-изоляцию и компонуемость
  3. Сохранять трассируемость версий flow
  4. Оптимизировать удобство имплементации только после структурной целостности

Использовать bounded review loops: 3–5 проходов для нестабильных задач, фокусные оптики, severity-triage и явные stop-критерии.

6. Критерии готовности (Definition of Done)

  • План Kanban существует и отслеживается по статусам.
  • Затронутые границы явно обновлены.
  • Инструменты остаются независимо исполняемыми.
  • Есть верификационные evidence-артефакты.
  • Проходит active-flow non-send E2E.
  • Выполнен coverage threshold (>= 80% total).
  • Evidence включает сводку тестов, сводку покрытия и путь к run-артефактам.

7. Анти-паттерны

  • Implementation-first без обновления протоколов/контрактов.
  • Скрытое поведение вне flow и манифестов.
  • Неявная связность между инструментами.
  • Недокументированное CLI-поведение.
  • Неверсионируемые изменения оркестрации.
  • Устаревшие legacy-инструкции, противоречащие активной runtime-цепочке.
  • Противоречивые mode-semantics в управляющих документах.

8. Шкала критичности (Severity Taxonomy, P0-P4)

  • P0: критический сбой корректности/безопасности; блокирует релиз.
  • P1: высоко-значимый сбой в core-flow; исправляется до релиза.
  • P2: существенный gap качества/надёжности; текущий или следующий цикл.
  • P3: умеренное улучшение; backlog при отсутствии немедленного риска.
  • P4: опциональная полировка.

Базовое release-правило: релиз блокируется при открытых P0/P1.

9. Итог

Золотой стандарт CineTaste — markdown-центричная, contract-driven, CLI-компонуемая архитектурная дисциплина. Это не только pipeline рекомендаций, но и контролируемая среда для воспроизводимой AI-assisted инженерной практики.
